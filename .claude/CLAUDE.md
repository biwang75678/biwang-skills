# 项目结构与需求开发流程规范

## 1) 仓库概览

- **仓库根目录**：`biwang-skills/`
- **远端仓库**：GitHub `biwang/biwang-skills`（origin）
- 本仓库为单项目仓库，所有内容围绕 biwang-skills 展开，不再按子项目划分。

## 2) 项目目录结构

```
biwang-skills/
├── .claude/
│   ├── CLAUDE.md              # 本规范文件
│   └── SKILL-DEV-GUIDE.md     # Skill 开发规范（dev-plans、交互流程、部署）
├── feat-dev/                  # 开发 feature vibe 记录（不入库）
│   └── YYYY-MM-DD/
│       ├── 01-xxx/
│       └── 02-yyy/
├── deploy/                    # 部署脚本和配置
│   ├── deploy.sh              # 主部署脚本
│   └── openclaw-plugin.json   # OpenClaw 插件 manifest 模板
├── skills/                    # Skill 源码
│   └── {skill-name}/
│       ├── SKILL.md           # Skill 定义入口（必须）
│       ├── dev-plans/         # 开发计划记录（入库）
│       ├── scripts/           # 脚本（可选）
│       ├── references/        # 参考文档（可选）
│       └── templates/         # 模板（可选）
└── ...
```

### Skill 开发规范

Skill 的创建和修改须遵循 `SKILL-DEV-GUIDE.md` 中的规范，核心要求：
1. 先在 `skills/{skill}/dev-plans/` 创建 plan 文件
2. 按交互流程与用户确认后实施
3. 提交 push 后执行 `deploy/deploy.sh` 部署

## 3) 每日需求目录管理规则

### 日期目录
- 在 `feat-dev/` 下按当天日期创建目录，格式为 `YYYY-MM-DD/`（如 `2026-03-16/`）。
- 每天首次接到需求时，若当天目录不存在则自动创建。

### 需求子目录命名
- 格式：`{序号}-{feature_description 极简概述}/`
- 序号为两位数字，从 `01` 开始递增。
- 创建前**必须先检查当天目录中已有的最大序号**，在此基础上 +1。
- 极简概述使用英文短横线连接的关键词（kebab-case），简明扼要。

示例：
```
2026-03-16/
├── 01-cli-init-research/
├── 02-skill-migration/
└── 03-command-scaffold/
```

## 4) 需求产出文档规范

每个需求目录下根据需求类型输出对应文档：

| 场景 | 产出文件 | 说明 |
|------|----------|------|
| 纯调研 | `research.md` | 调研结论、技术选型、参考资料等 |
| 基于调研的开发 | `plan.md` | 引用/依赖已有的 `research.md`，列出开发计划 |
| 调研 + 开发计划一体 | `plan.md` | 文档内先写调研内容（Research 部分），紧接列出开发计划（Plan 部分） |

### 需求原文结构化整理（固定步骤）
- 每次接到用户的需求描述后，**必须**将用户原始输入的需求内容进行语言和表达上的结构化整理。
- 整理后的内容放在 `research.md` 或 `plan.md` 文档的**最开头位置**，作为独立章节，标题为 `## 需求描述（整理）`。
- 整理要求：保留用户原始意图，但在语言表达上做清晰化、结构化处理（分点、分层、去冗余、补充隐含信息），使需求描述更易于理解和回顾。

### 文档编写要求
- 使用中文撰写，代码/API/术语保留英文。
- 结构清晰，使用分级标题、列表、表格、代码块。
- 是否进入实际编码开发，由用户指令决定，不主动执行。

## 5) 工作流程（接到需求时）

1. 确认今日日期，检查 `feat-dev/{YYYY-MM-DD}/` 是否存在，不存在则创建。
2. 检查当天目录下已有需求子目录的最大序号。
3. 创建新需求目录：`{序号+1}-{极简概述}/`。
4. 根据需求类型输出 `research.md` 和/或 `plan.md`。
5. 等待用户后续指令决定是否进入开发阶段。
